## Architecture Decisions Made

### 1. VerificationStatus as Enum Cast
The `verified` column on Photo is now cast to a `VerificationStatus` BackedEnum. This means:
- `$photo->verified` returns an enum instance, not an int
- Use `->value` for ordering comparisons (`>=`, `<`)
- Use direct enum identity (`===`) for equality
- Laravel auto-extracts `->value` in Eloquent `where()` clauses
- `json_encode()` serializes to the backing int value

### 2. `geom` Column Hidden from JSON
The `geom` column contains binary spatial data (`ST_GeomFromText(...)`) that cannot be UTF-8 encoded. Added to Photo `$hidden` to prevent JSON serialization errors in all API responses.

### 3. Team Type Resolution via Relationship
`Team::getTypeNameAttribute()` queries `teamType()` relationship instead of hardcoding ID-to-name mappings. This ensures correct behavior with `RefreshDatabase` in tests (where IDs depend on insertion order).

### 4. School Approval as MetricsService Trigger
For school teams, `TeamPhotosController::approve()` is the point where `TagsVerifiedByAdmin` fires — not `AddTagsToPhotoAction`. This means teacher approval IS the verification event. Summary/XP must be generated at tagging time regardless of trust level, or they'll be null at approval time and MetricsService will extract zero.

### 5. Tag Editing Resolves Strings to FKs
`TeamPhotosController::updateTags()` accepts category/object as string keys (`smoking`, `cigarette_butt`) and resolves them to `category_id`/`litter_object_id` via `Category::where('key', ...)` and `LitterObject::where('key', ...)`. This matches the API contract expected by the frontend.

## Session 3: Mobile Tag Shim — ConvertV4TagsAction

### Problem

Mobile apps send v4 tag format `{smoking: {butts: 5}}` which the old `AddTags` job writes to deprecated category table columns. After v5 migration, these photos get no `photo_tags`, no `summary`, no `xp` — they're invisible to MetricsService and all location totals. Mobile users can't contribute to LitterWeek team metrics.

### Approach: Reuse Migration Pipeline

Instead of building a new v4→v5 format converter, `ConvertV4TagsAction` reuses the same pipeline as the `olm:v5` migration script:

1. **Old action writes v4 data to category columns** — `AddTagsToPhotoAction` (in `App\Actions\Photos`) creates rows in `smoking`, `food`, etc. tables
2. **`UpdateTagsService::updateTags()`** reads category data back via `$photo->tags()`, classifies each tag via `ClassifyTagsService`, creates v5 `PhotoTag` + `PhotoTagExtraTags`, generates summary + XP
3. **Verification** — trusted users get `TagsVerifiedByAdmin` dispatched (→ MetricsService pipeline)

### Files Changed (4 files: 2 new, 2 modified)

| File | Change |
|------|--------|
| `app/Actions/Tags/ConvertV4TagsAction.php` | **New** — orchestrates old write → UpdateTagsService → verification |
| `app/Http/Controllers/API/AddTagsToUploadedImageController.php` | Replaced `AddTags` job dispatch with `ConvertV4TagsAction` (synchronous) |
| `app/Http/Controllers/ApiPhotosController.php` | Same — replaced `AddTags` dispatch in `uploadWithOrWithoutTags()` |
| `tests/Feature/Mobile/ConvertV4TagsTest.php` | **New** — 7 tests |

### ConvertV4TagsAction Details

- **Idempotency:** Checks `migrated_at` and `photoTags()->exists()` before processing
- **Unknown categories:** Filtered against `Photo::categories()` (hardcoded list of known category keys). Unknown keys logged and skipped — prevents crash from missing Photo relationship methods.
- **Remaining/picked_up:** Set before summary generation so `GeneratePhotoSummaryService` can apply XP bonus
- **Verification:** Reuses same logic as `AddTagsToPhotoAction` (v5) — `verification_required` check, school team detection, `TagsVerifiedByAdmin` dispatch with full constructor args
- **JSON string handling:** Controllers decode `is_string()` tags/custom_tags before passing to action (mobile may send JSON strings)

### Controller Changes

**AddTagsToUploadedImageController** (`POST /api/v2/add-tags-to-uploaded-image`):
- Removed `use App\Jobs\Api\AddTags` import
- Replaced `dispatch(new AddTags(...))` + separate `$photo->remaining` save with single `ConvertV4TagsAction::run()` call
- Response message changed from `'dispatched'` to `'tags-added'` (now synchronous)

**ApiPhotosController** (`uploadWithOrWithoutTags`):
- Removed `use App\Jobs\Api\AddTags` import
- Replaced `dispatch(new AddTags(...))` with `ConvertV4TagsAction::run()`
- Uses `!$photo->remaining` for pickedUp (already calculated in `storePhoto()`)

### Architecture Decision: Reuse vs. Rebuild

The Mobile.md design doc proposed a standalone converter using `ClassifyTagsService` directly. Instead, we reuse `UpdateTagsService::updateTags()` — the same code path as the migration script. Benefits:

1. **Battle-tested** — migration script already converted 500k+ photos with this code
2. **Handles all edge cases** — deprecated tag normalization, brands-only photos, custom tags, materials from deprecated mappings
3. **Single source of truth** — if classification logic changes, both migration and mobile shim benefit
4. **Less code** — ConvertV4TagsAction is ~100 lines (orchestration only), not ~200 lines (full conversion logic)

Tradeoff: writes to deprecated category tables first, then reads them back. Extra DB writes to tables that will be dropped post-migration. Acceptable — correctness over efficiency for a temporary shim.
