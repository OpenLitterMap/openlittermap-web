### Priority 3: Fixed `result_string` not populated for v5 photos

**Bug discovered:** Map endpoints (`GlobalMapController`, `DisplayTagsOnMapController`, `TeamsClusterController`, `GetPaginatedHistoryController`) all read `photo.result_string` for GeoJSON display and tag filtering. The v5 pipeline (`GeneratePhotoSummaryService`) did NOT set `result_string` — only `summary` JSON. New v5 photos would show null tags on all maps.

**Fix:** Added `result_string` generation to `GeneratePhotoSummaryService::run()`. Iterates `$sortedGrouped` and `$keyMap` to produce v4-compatible format: `category.object qty,category.object qty,...` (e.g., `smoking.butts 3,food.wrapper 2,`).

The migration script (`UpdateTagsService`) calls `GeneratePhotoSummaryService::run()`, so migrated photos will also get `result_string` populated.

**File:** `app/Services/Tags/GeneratePhotoSummaryService.php`

### Priority 4: Fixed PHP reference bug in GeneratePhotoSummaryService

**Bug:** Adding the `result_string` generation exposed a latent PHP reference bug. The existing code used `foreach ($objects as $objId => &$objData)` (line 216) to clean empty arrays, but never called `unset($objData)` after the loop. The dangling reference caused subsequent `foreach` loops using the same variable name to corrupt array data via the stale reference.

**Symptom:** `CalculatePhotoXpTest::multiple_objects_in_same_category_are_sorted_desc` failed — quantities `[3, 1]` became `[3, 3]` because the second foreach overwrote the last element via the reference.

**Fix:** Added `unset($objData)` after the `&$objData` reference foreach (line 227) and `unset($map)` after the `&$map` reference foreach (line 248).

**File:** `app/Services/Tags/GeneratePhotoSummaryService.php`

### Task 2: Fixed WebDeletePhotoTest (5 failures → 5 passing)

**Root cause:** Tests uploaded photos via `POST /api/photos/submit` (uses `auth:api` middleware) but authenticated with `actingAs($user)` (web guard, from Session 14's auth migration). The upload silently returned 401, no photo was created, and `$user->fresh()->photos->last()` returned `null`.

**Fix:** Rewrote all 5 tests to use `Photo::factory()` instead of uploading through the API. These tests validate deletion behavior, not upload — factories are the correct approach.

**File:** `tests/Feature/Photos/WebDeletePhotoTest.php`

### Task 3: Auth guard audit — 3 route fixes

Audited all profile/settings/user-photo routes for auth guard consistency.

| Issue | Severity | Fix |
|-------|----------|-----|
| `GET /api/user` — no middleware, calls `Auth::guard('api')->user()` | **Critical** — crashes if unauthenticated | Added `->middleware('auth:api')` |
| `DELETE /api/photos/delete` — no middleware | **Critical** — anyone can delete photos | Added `->middleware('auth:api')` |
| `PhotosController` constructor has `$this->middleware('auth')` conflicting with route's `auth:sanctum` | Low — redundant middleware | Removed constructor middleware |

**Architecture confirmed:** Settings routes are intentionally split across two auth groups:
- `auth:sanctum` — SPA endpoints (ApiSettingsController, DeleteAccountController, privacy toggles)
- `auth:api` — Legacy mobile endpoints (UsersController@details, changePassword, togglePrivacy, phone)

These are NOT duplicates — they're separate controllers serving different clients. Do not merge them.

**Files:** `routes/api.php`, `app/Http/Controllers/PhotosController.php`
