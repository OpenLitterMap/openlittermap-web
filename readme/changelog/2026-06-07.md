## Session: Custom tag sanitization (500 fix)

- `AddTagsToPhotoAction::attachCustomTags()` no longer throws `\Exception('Invalid custom tag.')` on punctuation. The allowlist regex `/^[\w\s:-]+$/` rejected legit brand/product chars (e.g. `&` in "Black & Mild"), and since the throw lived inside `run()`'s `DB::transaction`, one cosmetically-bad custom tag 500'd the request AND rolled back the user's valid object tags. Replaced with sanitize-and-accept: `mb_substr(trim(strip_tags($key)), 0, 255)` (caps to `custom_tags_new.key` varchar(255)); empty-after-sanitize tags are skipped (`continue`), never thrown. Applies to both standalone (`createExtraTagOnly`) and object-attached (`createTagFromClo`) custom tags. `bn:`→brand resolution intentionally left for the separate "brands later" ticket — `bn:` tags store as literal custom strings. (v5.11.6)
- Tests: added 4 to `AddNewTagsToPhotosTest` — `&` saves, `. ' /` save, object+bad-custom both persist (no rollback), empty custom skipped not thrown. (v5.11.6)
- Output-path security review (custom tags are now free text): stored XSS confirmed safe across all render paths — map popups escape via `htmlSanitizer.escapeHtml` (textContent→innerHTML), Vue uses `{{ }}`, the one `v-html` (tag-search highlight) pre-escapes, API returns JSON, and custom tags never reach the Twitter impact reports. (v5.11.6)
- CSV export: decided to keep cell values **byte-exact** (no formula-injection prefix). PhpSpreadsheet quoting already prevents delimiter/quote/newline breakout; a `=+-@` prefix would corrupt programmatically-consumed research data (e.g. negative longitudes) for a warning-gated spreadsheet-only risk. Documented the verbatim caveat in `readme/ExportData.md` (treat `custom_tag_*`/`model`/`display_name` as untrusted when opening exports in a spreadsheet app) and logged a backlog ticket for a string-typed `.xlsx` export as the proper spreadsheet-safe path. (v5.11.6)
- Docs sync: deleted the working `readme/upload-spec.md`; corrected `readme/API.md` `custom_tags` rule (free text, 255-cap, not "max 100"); completed `readme/Upload.md` (POST already-tagged guard + PUT onboarding parity); added the idempotent duplicate-upload response to `readme/Mobile.md`; updated `readme/Onboarding.md` (PUT `update()` is now a third onboarding-completion trigger). Skills: `tagging-system` (custom-tag sanitization invariant). (v5.11.6)

## Session: HEIC upload fix

- Fixed HEIC photo upload (two bugs, both confirmed against production). **Bug 1:** `MakeImageAction` shelled out to `magick convert`, but the server has ImageMagick 6 (`convert`, no `magick`) — every HEIC conversion failed. Hard-switched to IM6 `convert input output`. **Bug 2:** `UploadPhotoRequest`'s `image` rule (excludes HEIC) and `dimensions` rule (`getimagesize()` returns false for HEIC) rejected genuine HEIC before it reached the converter. `rules()` now drops `image`+`dimensions` for HEIC (detected via the now-public `MakeImageAction::isHeic()` — magic-byte aware, catches iOS HEIC sent as `.jpg`), keeping `mimes`+`max`. Why "some users": only those sending genuine HEIC bytes failed; iOS auto-transcoded-to-JPEG uploads always worked. (v5.11.7)
- Test: added `HeicUploadTest::test_genuine_heic_uploads_successfully` — posts a real committed HEIC fixture (`storage/framework/testing/sample.heic`) to `/api/v3/upload` (mobile mode) and asserts success, with `MakeImageAction` faked (no `convert` binary locally/CI; Bug 1 verified on production). The prior HeicUploadTest only reflection-tested `isHeic()`, never the endpoint. (v5.11.7)
- Follow-up ticket (not implemented): web-mode HEIC (no explicit lat/lon/date) still hits `after()`'s `exif_read_data`/GPS checks, unreliable on HEIC. Mobile path fully fixed. (v5.11.7)
- Docs: `readme/Upload.md` (new HEIC handling section + fixed stale mobile duplicate-422 line), `readme/API.md` (`photo` rule note), `readme/Mobile.md` (HEIC supported). Skills: `photo-pipeline` (HEIC validation-skip + `convert`-not-`magick` invariant + Common Mistake). (v5.11.7)

## Session: Login throttle fix (too strict for shared IPs)

- Fixed over-aggressive login throttling reported by real users. The old `POST /api/auth/login` used Laravel's generic `throttle:5,1` middleware, which (a) counted **successful** logins toward the limit and (b) keyed by **IP only** — so a whole classroom/office behind one NAT shared a 5/min budget and got 429s with correct credentials. Replaced with a controller-level limiter in `LoginController`: keyed by `identifier + IP`, **only failed** attempts count (success calls `RateLimiter::clear`), 7 failures/min before a 429, still disabled on localhost. Removed `throttle:5,1` from the route (now just `['web']`). (v5.12.1)
- Tests (`LoginTest`): rewrote the rate-limit test to 7 failures→429, added `test_successful_logins_are_not_rate_limited` (12 consecutive successes all 200) and `test_one_account_being_throttled_does_not_lock_out_another` (locked account 429s while a different identifier on the same IP logs in fine). 9 passing. (v5.12.1)
- Docs: `readme/API.md` login section documents the new identifier+IP, failures-only limit. (v5.12.1)
