# Changelog — 2026-06-12

## Session: HEIC upload failure (iOS 18 / server libheif)

- Diagnosed `/api/v3/upload` HEIC failures (`MakeImageAction.php:167`, "ImageMagick returned code 1") as a **server-side libheif issue**: libheif < 1.18.2 cannot decode iOS 18+ HEIC ("Metadata not correctly assigned to image @ heic.c/IsHEIFSuccess"). Not an app bug — the conversion code is correct; the server's libheif is too old. (v5.12.2, pending patch bump on BOOP)
- Added `App\Exceptions\HeicConversionException`; `MakeImageAction` now throws it on conversion failure. `UploadPhotoController` catches it and returns a graceful typed **422** `{ error: 'heic_conversion_failed' }` instead of a hard 500, so affected mobile users get a friendly message. (v5.12.2)
- Test: `HeicUploadTest::test_heic_conversion_failure_returns_graceful_422` — asserts 422 + typed error + no photo created. Full HEIC suite (14 tests) green. (v5.12.2)
- Docs: `readme/Upload.md` HEIC section now documents the libheif root cause + a Forge server upgrade runbook (≥ 1.18.2) + mobile-app follow-up (convert HEIC→JPEG client-side); `readme/API.md` upload error table now lists `heic_conversion_failed` and `geocoding_failed`. (v5.12.2)

## Session: Switch HEIC conversion from ImageMagick `convert` to `heif-convert`

- **Superseded** the prior libheif-upgrade conclusion. Production diagnosis (Ubuntu 24.04, libheif 1.17.6-1ubuntu4.3, IM6) showed the IM6 HEIC delegate is unreliable across HEIC variants: 2026 Xiaomi HEIC (`mif1, MiHE, MiPr, miaf, MiHB, heic`) fails `convert` with `no encode delegate for this image format 'HEIC'`, while `heif-convert` (from `libheif-examples`, already installed on prod) succeeds on every sample. Empirically verified `heif-convert -q 92` exit 0 + GPS-intact output on the real Xiaomi sample. (v5.12.3)
- `MakeImageAction`: replaced the `exec('convert …')` shell-out with the `Process` facade running `heif-convert -q 92 {input} {output}` (array command, 60s timeout). Renamed `convertViaImageMagick` → `convertViaHeifConvert`; single conversion path, no ImageMagick fallback. `heif-convert` embeds EXIF/GPS/XMP/ICC and bakes orientation by default (existing `->orientate()` is now a no-op); `-q 92` matches the old IM quality. (v5.12.3)
- Failure preservation: on non-zero exit or missing output, the source HEIC is **moved** to `storage/app/heic_failed/{hash}.heic` (diagnostic sample), `Log::error` records `original_name`/`exit_code`/combined stdout+stderr/`preserved_path` (Sentry), then `HeicConversionException` → controller 422. `finally{}` skips preserved files. New constants: `FAILED_HEIC_STORAGE_DIR`, `HEIC_CONVERT_TIMEOUT=60`, `JPEG_QUALITY=92`. (v5.12.3)
- Test: added `HeicUploadTest::test_heic_conversion_process_failure_preserves_sample_and_returns_422` — drives the real `heif-convert` path with `Process::fake()` (exit 1), asserts 422 + `heic_conversion_failed`, a preserved `.heic` in `heic_failed/`, the temp file gone from `heic_images/`, and no Photo created. Full HEIC suite (15 tests) green; upload + photo-API suites (50 tests) green. (v5.12.3)
- Docs: rewrote `readme/Upload.md` HEIC section for the `heif-convert` path (both Sentry signatures, EXIF/orientation behaviour, failure-preservation contract, `heic_failed/` ops notes, `libheif-examples` server dependency); marked the **422-on-failure behaviour as INTERIM** — target is accept-on-failure (store original HEIC, flag it, reprocess via a future `olm:reprocess-failed-heic` command). `readme/API.md` error-table entry updated + flagged interim. (v5.12.3)
